✅Explain your SpringShield project.

ANSWER:
SpringShield is a Spring Boot-based REST API security project where I implemented stateless authentication using JWT.

The system supports login using username and password, which are securely stored using BCrypt hashing. After successful authentication through Spring Security’s AuthenticationManager, the system issues an access token and a refresh token.

The access token is short-lived and used to secure protected endpoints. I implemented a custom JWT filter that intercepts each request, validates the token, and sets the authentication in the SecurityContext.

Additionally, I enabled method-level security using @PreAuthorize to enforce role-based access control at the service layer, ensuring fine-grained authorization.

I also implemented a database-backed refresh token mechanism to allow secure token renewal without requiring users to log in again.



<--------------------------------------------------------------------------------------------------------------->
✅Explain authentication flow in your project.

ANSWER:
At login, AuthenticationManager validates credentials using CustomUserDetailsService and BCrypt. Upon success, JWT access and refresh tokens are issued. For protected endpoints, the JwtAuthenticationFilter intercepts requests, validates the access token, and sets the authentication in the SecurityContext. Spring Security then performs authorization checks before allowing access to secured methods.

The 10x improvement refers to the granularity of access control we achieved. Before implementing RBAC and method-level security, access control was binary — a user was either authenticated or not, and once authenticated they could call any service. After the implementation, every service method had its own access control via @PreAuthorize annotations, different roles had different permissions, and even within the same role we could control access at a per-user level.

⭐Before (no security / basic security):

No distinction between roles
One user could access everything another user could
Access control was 0 or 1 — binary

⭐After (RBAC + Method-level security):
Control at endpoint level via @PreAuthorize
Control at method level inside services
Multiple roles like ADMIN, USER, MODERATOR etc.
Each role has different permissions on different methods
Same endpoint can behave differently based on role

✅What is this 15% improvement and how did you achieve it?

ANSWER:
The 15% performance improvement came from eliminating the overhead of repeated full authentication flows. Before implementing refresh tokens, when an access token expired the user had to re-login, which triggered a full authentication cycle including BCrypt password comparison, which is intentionally slow for security reasons, plus multiple HTTP redirects. After implementing automated refresh token functionality, token renewal happens silently in the background with just a lightweight database lookup.

⭐Before (without refresh token):
Access token expires → user gets 401 Unauthorized
User has to manually re-login
New login = full authentication round trip:
Credentials sent to server
Password BCrypt comparison (intentionally slow)
New token generated
Extra HTTP requests


⭐After (with automated refresh token):
Access token expires → refresh token automatically sends a request
Server validates refresh token (just a DB/cache lookup — very fast)
New access token issued silently
No BCrypt comparison, no credential transmission, no user interruption
Fewer round trips, lighter auth endpoint load

✅What is the difference between URL-based security and method-level security in Spring Security, and why did you choose method-level security?

ANSWER:
URL-based security enforces authorization rules at the HTTP endpoint level using request matchers in SecurityConfig. It protects routes but does not secure the underlying business logic directly.

Method-level security, enabled using @EnableMethodSecurity and annotations like @PreAuthorize, enforces authorization at the service layer. This ensures that even if a method is invoked internally or reused elsewhere, authorization checks are still applied.

I chose method-level security to achieve fine-grained control and ensure that business logic is protected independently of the web layer.

✅Why is it important to make the application stateless when using JWT, and what would happen if you used session-based authentication together with JWT?

ANSWER:
JWT-based authentication is designed to be stateless because all user identity and authorization data are embedded within the token itself. This eliminates the need for server-side session storage. If session-based authentication were used alongside JWT, it would defeat the purpose of token-based stateless authentication by reintroducing server-side state, reducing scalability, and increasing complexity.

✅Why is BCrypt preferred over storing hashed passwords using SHA-256 or MD5?

ANSWER:
SHA-256 and MD5 are designed for fast hashing and are vulnerable to brute-force and rainbow table attacks when used for password storage. BCrypt is specifically designed for password hashing. It includes built-in salting and an adjustable work factor, making it computationally expensive to crack passwords through brute force attacks.




<--------------------------------------------------------------------------------------------------------------->
✅Why is it recommended to store refresh tokens in the database but not store access tokens in the database?

ANSWER:
Access tokens are short-lived and self-contained JWTs designed for stateless authentication, so storing them in the database would defeat their performance and scalability benefits. Refresh tokens, however, are long-lived and effectively act as re-authentication credentials, so they must be stored in the database to allow revocation, logout support, and protection against token misuse.

✅What did you improve compared to a basic Spring Security setup?

ANSWER:
Initially, the system relied on default Spring Security configuration using HTTP Basic authentication and session-based state management.

I migrated it to stateless JWT-based authentication to remove server-side session storage and improve scalability. Stateful authentication system are not easily scalable.

I also introduced role-based authorization at the method level, ensuring business logic is protected independently of HTTP routes.

Additionally, I implemented refresh tokens stored in the database to support secure token renewal and logout functionality.
